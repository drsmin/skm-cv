사용자 행동 기록 성능 향상

1

문제 발생

시스템에 사용자가 몰릴 시 기대 했던 것 보다 시스템 부하가 심각하게 발생
테스트에서는 발견되지 않았지만 오픈 후 서비스가 몰릴 때 발생
2

원인

사용자의 거래 기록을 전수로 DB에 기록하는 행위가 사용자가 급증 시 부하를 가중 시킴
3

해결

- 거래 기록의 Key 채번을 Lock-select 가 아닌 Sequence로 변경
- 사용자의 거래 기록을 Queue에 저장 그 후 일정 시간 마다 Batch Statement 처리를 이용하여 처리함.
- 일정 처리 시간안에 해당 기록을 처리하지 못하면 유동적으로 처리 Thread를 늘려 다중으로 처리함
- 거래 기록을 RAW데이터가 아닌 매핑 처리 하여 전체적인 데이터 처리량을 줄임

대외 연계 성능 향상

1

문제 발생

은행연합회 대외 처리가 너무 느림
단순 처리에는 문제가 없었지만 일마감 작업등에 담당자가 고통을 호소
2

원인

은행 연합회와의 통신 방식이 모두 동기화 식으로 건 by 건 동기식으로 처리 됨
3

해결

- Netty 라이브러리를 통해 비동기 방식으로 변경
- 인터페이스 원장 처리 Batch Statement로 변경

Nexacro 화면 렌더링 속도 저하

1

문제 발생

기존 X-Platform에서 사용하던 화면이 Nexacro로 변환 하였더니 속도가 매우 느려짐
기존 1초 -> 2분으로 도저히 정상 사용이 불가능 함
2

원인

Grid 의 렌더링 방식이 row / column이 생성 될 때마다 모든 처리를 반복함 (렌더링 및 이벤트 리스너 까지 재 계산)
3

해결

- Grid의 모든 속성 및 이벤트를 Object로 저장
- Grid 렌더링
- 렌더링 이후 원래 속성 및 이벤트를 일괄 적용
- 위의 과정을 공통화 하여 적용

Java 시스템 OOM

1

문제 발생

시스템에서 간헐적인 Out Of Memory 에러가 발생
2

원인

어플리케이션에서 다량의 데이터를 한번에 Select 하여 가져옴
3

해결

- Java Memory Analyzer 를 이용해 dump 파일을 분석
- 특정 List객체에 1.7G의 데이터가 메모리에 할당된 것을 찾음
- 해당 기능은 Mybatis 의 rowHandler를 이용하여 일정 데이터 이상이면 화면으로 보내고 화면에서는 해당 데이터를 append 형식으로 표현
- 위의 기능을 표준화 하여 프로젝트 팀에 가이드

대량의 센서 데이터 처리

1

문제 발생

초당 3,000건 이상의 IoT센서의 데이터를 분석하는데 시간이 흐를수록 DB 적재 처리가 밀려서 처리가 실시간이 되지 않는 경우가 생김
2

원인

DB의 처리 능력이 초당 3,000건 이상을 처리하지 못함
3

해결

- 최초의 로우 데이터 기록을 In Memory H2 DB에 기록함
- In Memory 데이터 중 무의미한 데이터 버림
- 유의미한 데이터만을 가지고 분석 처리
- 유의미한 데이터 및 분석 결과를 실제 DB에 적재
